1. What is the Traveling Salesman Problem?

TSP Decision asks whether there is one closed route through all cities whose total cost is at most a given bound B: start at a city, visit every other city exactly once, and return to the start.

Hint: A cycle is a route that comes back to where it started; a TSP tour is a cycle through all cities.

Real-world idea
Delivery routes, school bus routes, warehouse picking paths, and road-trip planning all have the same flavor: many stops, one route, limited travel.
Computer science idea
TSP is a model for sequencing tasks: order the visits so the total cost is small. The “cities” can be physical locations, jobs, or states in a system.
Why it is hard
The number of possible tours grows very quickly. A route problem that looks small can become too large for exact checking.

2. The decision version of TSP

TSP is often written in two forms. The optimization version asks: “what is the cheapest possible tour?” The decision version asks a yes/no question instead: “is there a tour cheap enough for this given bound?” NP-completeness proofs use the decision version because complexity classes like NP are about yes/no problems.

So in this tool, the input gives a complete weighted graph and a bound B. We ask: is there a TSP tour with total cost ≤ B? A valid TSP tour has Hamiltonian structure: it visits every city exactly once before returning to the start.

Key Concept: Instance
One full input to the decision problem: a complete weighted graph on n cities plus a bound B.
Key Concept: Certificate
A proposed TSP tour, such as A → C → D → B → A.
Key Concept: Verifier
Check the list length, make sure every city appears exactly once, add the tour edge weights, and compare the total with B.

3. Why exact search grows so fast

The most direct exact algorithm is brute force. It solves TSP by trying every possible city order, computing the full tour cost, and keeping the best one. That is why it is useful as the truth benchmark: if it checks every tour, the best one it finds really is optimal.

But the price is growth. If we fix the starting city, there are roughly (n - 1)! tours. This is why the tool includes a growth chart: for small n, exact solving feels fine; for larger n, the number of tours becomes the main story.

4. Heuristics are helpful, but not proof

When exact search becomes too expensive, we use heuristics: practical algorithms that try to find good tours quickly. Nearest Neighbor is a greedy heuristic: from the current city, go to the closest unvisited city. Greedy + 2-Opt starts with that greedy tour, then improves it by swapping edges when a swap saves distance.

These methods are useful, but they are not proof. For the decision problem, a heuristic tour with cost greater than B does not prove the answer is no. Only an exact method or a formal proof can settle every case.

Key Concept: 2-Opt swap
Pick two non-adjacent tour edges, such as (a,b) and (c,d). Remove them, then reconnect as (a,c) and (b,d). This reverses the middle part of the route.
Key Concept: Delta
delta means change in cost: new edge cost - old edge cost. A negative delta means the new route is shorter.
Accept rule
If delta < 0, keep the swap and continue improving. If delta ≥ 0, reject it. Repeat until no improving swap is found.

5. Worked example: checking one certificate

Suppose the cities are A,B,C,D, the bound is B=1000, and someone gives the certificate A → B → C → D → A. The verifier does not search for a better tour. It only checks this proposed tour: every city appears exactly once, the return to A is included, and the four edge costs are added. If the sum is at most 1000, this certificate proves the instance is YES.

7. Counterexample idea: 2-Opt can stop too early

What 2-Opt is allowed to change
2-Opt is local search. It looks at two tour edges, removes them, reconnects the route, and keeps the change only if the tour gets shorter.
Where the trap happens
A tour can be 2-optimal, meaning no single 2-edge swap improves it, but still not be globally optimal. It is like standing in a small valley: every nearby step goes uphill, even though a deeper valley exists farther away.
What this teaches
2-Opt tests many local repairs, not every possible tour. So “no improving 2-Opt swap remains” means local optimum, not “proof of best possible tour.”

This local-search trap idea is discussed in the CS StackExchange explanation of why 2-Opt is not always optimal, and formal difficult examples are constructed by Papadimitriou and Steiglitz in Some Examples of Difficult Traveling Salesman Problems.

8. Advantages and disadvantages

Brute Force
Advantage: exact. It finds the true best tour for small n.
Disadvantage: factorial growth. After fixing the start city, it still checks (n-1)! tours.
Nearest Neighbor
Advantage: fast and simple greedy construction. It quickly gives a complete tour.
Disadvantage: local choices can trap it. A close next city may force an expensive route later.
Nearest Neighbor + 2-Opt
Advantage: improves the greedy tour by replacing two bad edges when the swap saves cost.
Disadvantage: still local search. Stopping means no 2-edge improvement remains, not proof of global optimum.

Dedicated NP-Completeness Proof Section

Formal language version
Notations used in this tab

G=(V,E): G is the graph, V is the set of vertices/cities, and E is the set of edges/connections.

|V|=n: the graph has n cities. w(u,v): the cost or distance between cities u and v. B: the maximum cost allowed for a YES answer.

Problem P: TSP-DECISION = {⟨G,w,B⟩ : G is a complete graph and there exists a TSP tour T with cost(T) ≤ B}.

Here G=(V,E), |V|=n, and w(u,v) is the edge cost. A valid TSP tour has Hamiltonian structure: start at one city, visit every city exactly once, then return to the start.

Plain-English version

We are not asking “what is the shortest tour?” yet. The decision question is simpler: is there at least one valid tour whose total cost is at most B? That yes/no form is what belongs in NP-completeness proofs.

Input
Complete weighted graph + bound B.
Output
YES if a cheap enough TSP tour exists; otherwise NO.
Step 1: TSP Decision is in NP
Notations used in this tab

G=(V,E): the complete weighted TSP graph. V is the city set, E is the edge set, and n=|V| is the number of cities.

w: the weight function used to look up edge costs. B: the decision bound. C: the certificate, meaning the proposed TSP tour we are verifying.

(a) Certificate format. Formally, a certificate is a proposed TSP tour written as an ordered sequence C = (c₁, c₂, ..., cₙ), where each cᵢ ∈ V. It claims the tour is c₁ → c₂ → ... → cₙ → c₁.

Example: if V = {A, B, C, D}, then C = (A, C, D, B) is a certificate claiming the TSP tour A → C → D → B → A. The return edge from the last city back to the first is implied.

The verifier checks length(C) = n first because a valid TSP tour must list every one of the n cities exactly once before returning to the start. If the list is too short, it skipped a city; if it is too long, it must repeat or include an invalid city.

(b) Verifier pseudocode.

VERIFY_TSP(G, w, B, certificate C) n = |V(G)| if length(C) != n: reject if C does not contain every vertex exactly once: reject total = 0 for i = 1 to n-1: total = total + w(C[i], C[i+1]) total = total + w(C[n], C[1]) if total ≤ B: accept else: reject

(c) Polynomial bound. The verifier checks one proposed tour, not every possible tour.

1. Length check
Check length(C)=n. This is O(1) if length is stored, and at worst O(n) if counted by scanning.
2. Vertex uniqueness
Scan the n certificate entries and use a boolean table or set to reject repeats. This is O(n).
3. Tour cost
Add the n tour edges, including the return edge. With matrix/table lookup, this is O(n).
4. Bound test
Compare total cost with B. This is O(1).

So the total verification time is O(n) using the usual adjacency-matrix/table representation, and therefore polynomial in the input size.

Intuition: We may not know how to find the best tour quickly, but if someone hands us a proposed tour, checking it is easy. That is exactly what “in NP” means.

Step 2: Reduction mapping f
Notations used in this tab

Q: the source problem, Hamiltonian Cycle. P: the target problem, TSP Decision. x=⟨G=(V,E)⟩: one input instance of Hamiltonian Cycle.

f: the reduction function. f(x)=⟨G′,w,B⟩: the TSP instance produced from x. V′=V, so the TSP graph uses the same vertices as the HC graph.

G′ is the complete graph on V′. It contains original edges from E with weight 1, plus complement/missing edges not in E with weight 2. w gives these edge weights, and B is the bound.

Source problem Q: Hamiltonian Cycle. Input x = ⟨G=(V,E)⟩. Question: does G contain a simple cycle that visits every vertex exactly once?

Target problem P: TSP Decision. We define a function f : Σ* → Σ* that converts an HC instance into a TSP instance.

f(⟨G=(V,E)⟩): V′ = V make G′ complete on V′ for every pair of distinct vertices u,v: if (u,v) is an edge in G: w(u,v) = 1 else: w(u,v) = 2 B = |V| output ⟨G′, w, B⟩
Hamiltonian Cycle input
A non-complete, unweighted graph G. We only know which edges exist.
→
Mapping f
Keep the same vertices. Existing edges become cost 1; missing edges become cost 2. Set B=n.
→
TSP Decision output
A complete weighted graph where tours using only original edges cost exactly n.
Concrete worked small instance

Take V={A,B,C,D} and original edges {AB,BC,CD,DA,AC}. The map creates the complete graph on the same four vertices. Edges AB,BC,CD,DA,AC get weight 1. Missing edge BD gets weight 2. Since n=4, set B=4. The cycle A-B-C-D-A uses four weight-1 edges, so the TSP instance is YES.

Polynomial-time construction
Work done by f
Check each pair of vertices and assign weight 1 if the edge existed in G, otherwise weight 2.
Time bound
There are n(n-1)/2 unordered pairs in the undirected proof, so the construction takes O(n²) time and outputs O(n²) weighted edges.

Intuition: Weight 1 means “this edge was legal in the original graph.” Weight 2 means “this edge was only added to make TSP complete.” The bound B=n forces any YES tour to avoid the added fake edges.

Correctness of the reduction
Notations used in this tab

G: the original Hamiltonian Cycle graph. f(G): the TSP Decision instance made from G. G′: the complete graph inside that TSP instance.

n=|V|: the number of vertices/cities. Original edges get weight 1, missing edges get weight 2, and the reduction sets B=n.

We must prove the exact equivalence:

G ∈ HC ⇐⇒ f(G) ∈ TSP-DECISION

  1. Forward direction, HC YES ⇒ TSP YES: Suppose G has a Hamiltonian cycle. A Hamiltonian cycle on n vertices uses exactly n edges. Every one of those edges is an original edge of G, so in G′ every one has weight 1. Total TSP tour cost is therefore n. Since B=n, the constructed TSP instance answers YES.
  2. Backward direction, TSP YES ⇒ HC YES: Suppose f(G) has a TSP tour of cost at most B=n. A tour visits every vertex and returns to the start, so it uses exactly n edges. Each edge has weight either 1 or 2. If even one edge had weight 2, the total would be more than n. So all chosen edges have weight 1, meaning all chosen edges existed in the original graph G. Those edges form a Hamiltonian cycle in G.
  3. Polynomial time bound of f: The construction checks every unordered pair of vertices. There are n(n-1)/2 pairs, so assigning all weights takes O(n²) time using an adjacency matrix or hash set for edge lookup. The output graph also has O(n²) edges, so this is polynomial in the input/output size.
  4. Conclusion: We have shown HC ≤p TSP-DECISION. Since Hamiltonian Cycle is NP-complete, this proves TSP Decision is NP-hard. Since TSP Decision is also in NP, TSP Decision is NP-complete.

Oral Q&A shortcut: “The whole trick is the bound B=n. A tour has n edges. If all are weight 1, cost is exactly n. One weight-2 edge already pushes the cost above n. So a cheap TSP tour is forced to be a Hamiltonian cycle in the original graph.”

Edge Cases & Clarifications
Notations used in this tab

G is the Hamiltonian Cycle input graph. G′ is the complete weighted TSP graph produced by the reduction. Existing edges get weight 1, missing edges get weight 2, and B=n.

If G is already complete
The construction still works. Every pair is already an original edge, so every TSP edge gets weight 1. A Hamiltonian cycle gives a tour of cost n.
If G has missing edges
That is the normal case. We add the missing edges only because TSP expects a complete graph. Those added edges receive weight 2, so they are too expensive for a tour with bound B=n.
If a tour uses one weight-2 edge
A valid TSP tour has exactly n edges. If one edge has weight 2 and the rest are at least 1, the total is greater than n. So it cannot be a YES tour under B=n.
If a certificate repeats a vertex
The verifier rejects it. A valid TSP tour must visit every vertex exactly once before returning to the start.
Very small graphs
The standard NP-completeness proof is for ordinary graph instances, usually n ≥ 3. Tiny cases can be answered directly and do not affect the polynomial-time reduction argument.
Are weights 1 and 2 allowed?
Yes. They are positive integer weights, so the constructed instance is a valid TSP Decision instance.

Oral Q&A shortcut: “The added weight-2 edges are not illegal; they are deliberately expensive. The bound B=n forces a YES tour to use only weight-1 edges, which means it was really a Hamiltonian cycle in the original graph.”

So Hamiltonian Cycle ≤p TSP Decision. Since Hamiltonian Cycle is NP-complete and TSP Decision is in NP, TSP Decision is NP-complete.
